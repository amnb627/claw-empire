## 1. 受入テスト概要

本ドキュメントは「Mobile Inbox & Watcher機能」の受入テスト基準を定義する。

---

## 2. テスト原則

| 原則                    | 説明                                       |
| :---------------------- | :----------------------------------------- |
| **合格 = リリース可能** | 全てのP0項目が合格ならばリリース可能と判断 |
| **失敗の明確化**        | 各テストケースに明確な失敗条件を定義       |
| **再現性確保**          | テスト手順は誰でも再現可能なレベルで記載   |
| **自動化優先**          | 回帰テストは可能な限り自動化する           |

---

## 3. Mobile Inbox 受入テスト基準

### 3.1 P0 (Critical) - ブロッカー

| ID            | テスト項目     | 合格条件                                                  | 失敗条件                                 |
| :------------ | :------------- | :-------------------------------------------------------- | :--------------------------------------- |
| **MI-P0-001** | モバイルUI表示 | 画面幅<640pxでスライドアップシートが表示                  | デスクトップモーダルが表示される         |
| **MI-P0-002** | アイテム取得   | `/api/decision-inbox` が200でアイテムを返す               | エラーまたは空配列（アイテムがある場合） |
| **MI-P0-003** | 意思決定送信   | 選択肢選択後、POST `/api/decision-inbox/:id/reply` が成功 | 4xx/5xxエラー                            |
| **MI-P0-004** | タッチ操作     | ボタン・カードのタッチが44px以上の領域で反応              | 44px未満のタッチ領域が存在               |
| **MI-P0-005** | データ永続化   | 決定結果がSQLiteに保存される                              | DB更新失敗                               |

### 3.2 P1 (High) - 機能要件

| ID            | テスト項目               | 合格条件                              | 失敗条件                         |
| :------------ | :----------------------- | :------------------------------------ | :------------------------------- |
| **MI-P1-001** | プル・ツー・リフレッシュ | 下スワイプで最新アイテムが取得される  | リフレッシュされない             |
| **MI-P1-002** | 右スワイプ承認           | 右スワイプで最初の選択肢が選択される  | アクションが実行されない         |
| **MI-P1-003** | 左スワイプ詳細           | 左スワイプで詳細が展開される          | 展開されない                     |
| **MI-P1-004** | ロングプレスチャット     | 長押しでチャット画面に遷移            | 遷移しない                       |
| **MI-P1-005** | スライドアニメーション   | シートが滑らかにスライドアップ/ダウン | カクつきまたはアニメーションなし |

### 3.3 P2 (Medium) - UX改善

| ID            | テスト項目         | 合格条件                             | 失敗条件                 |
| :------------ | :----------------- | :----------------------------------- | :----------------------- |
| **MI-P2-001** | 空状態表示         | アイテムなし時に適切なメッセージ表示 | 空白またはエラー         |
| **MI-P2-002** | バッジ更新         | 未読数バッジが正確に更新される       | カウント不正確           |
| **MI-P2-003** | タイムスタンプ表示 | 相対時間（「2分前」）が正確          | 不正確または絶対時間のみ |

## 4. Watcher機能 受入テスト基準

### 4.1 P0 (Critical) - ブロッカー

| ID            | テスト項目           | 合格条件                                   | 失敗条件           |
| :------------ | :------------------- | :----------------------------------------- | :----------------- |
| **WA-P0-001** | 監視登録             | タスク/プロジェクトの監視開始トグルが動作  | トグルが反応しない |
| **WA-P0-002** | 通知配信             | 監視対象のステータス変更時に通知が送られる | 通知が来ない       |
| **WA-P0-003** | YOLOオートパイロット | YOLO有効時に自動決定が実行される           | 自動決定されない   |
| **WA-P0-004** | スキップ条件         | `video_preprod` パックは自動決定スキップ   | スキップされない   |

### 4.2 P1 (High) - 機能要件

| ID            | テスト項目    | 合格条件                                   | 失敗条件                 |
| :------------ | :------------ | :----------------------------------------- | :----------------------- |
| **WA-P1-001** | 通知優先度    | Critical/High/Normal/Lowで通知が分類される | 全て同じ優先度           |
| **WA-P1-002** | 監視一覧表示  | 監視中タスク一覧が正確に表示される         | 不正確または表示されない |
| **WA-P1-003** | WebSocket配信 | ステータス変更がリアルタイムに反映される   | 遅延または反映されない   |
| **WA-P1-004** | 通知設定変更  | 通知ON/OFFが即座に反映されない             | 反映されない             |

### 4.3 P2 (Medium) - UX改善

| ID            | テスト項目   | 合格条件                   | 失敗条件                       |
| :------------ | :----------- | :------------------------- | :----------------------------- |
| **WA-P2-001** | トースト通知 | 通知がトーストで適切に表示 | 表示されないまたは即座に消える |
| **WA-P2-002** | 通知履歴     | 過去の通知履歴が参照可能   | 履歴機能なし                   |

## 5. 統合テスト基準

### 5.1 P0 (Critical)

| ID             | テスト項目        | 合格条件                                   | 失敗条件           |
| :------------- | :---------------- | :----------------------------------------- | :----------------- |
| **INT-P0-001** | Inbox→Watcher連携 | Inbox内から監視開始が可能                  | 連携されない       |
| **INT-P0-002** | Messenger通知     | Telegram/DiscordでDecision Inbox通知が届く | 通知が来ない       |
| **INT-P0-003** | 既存機能互換      | 既存DecisionInboxModalが動作し続ける       | レグレッション発生 |

### 5.2 P1 (High)

| ID             | テスト項目       | 合格条件                              | 失敗条件                 |
| :------------- | :--------------- | :------------------------------------ | :----------------------- |
| **INT-P1-001** | レスポンシブ切替 | 画面サイズ変更でUIが切り替わる        | 切り替わらないまたは遅延 |
| **INT-P1-002** | データ整合性     | モバイル/デスクトップ間でデータ一貫性 | 不整合                   |

## 6. パフォーマンス基準

| メトリクス            | 目標値   | 測定方法                    |
| :-------------------- | :------- | :-------------------------- |
| **初期ロード時間**    | < 2秒    | Lighthouse / DevTools       |
| **アイテム取得**      | < 500ms  | APIレスポンスタイム         |
| **アニメーションFPS** | >= 60fps | Chrome DevTools Performance |
| **WebSocket遅延**     | < 100ms  | イベント発火→受信までの時間 |

## 7. セキュリティテスト基準

| ID          | テスト項目          | 合格条件                          | 優先度 |
| :---------- | :------------------ | :-------------------------------- | :----- |
| **SEC-001** | API認証             | 認証なしのAPIアクセスが拒否される | P0     |
| **SEC-002** | SQLインジェクション | ユーザー入力で安全なクエリ実行    | P0     |
| **SEC-003** | XSS防止             | ユーザー入力がエスケープされる    | P1     |
| **SEC-004** | CSRF保護            | POSTリクエストにトークン検証      | P1     |

## 8. 非機能テスト基準

### 8.1 ユーザビリティ

| 項目               | 基準                |
| :----------------- | :------------------ |
| **タッチ領域**     | 最小44×44px         |
| **コントラスト比** | WCAG AA以上 (4.5:1) |
| **フォントサイズ** | 最小14px            |

### 8.2 互換性

| プラットフォーム | 対象バージョン |
| :--------------- | :------------- |
| iOS Safari       | 15+            |
| Chrome Mobile    | 最新版 ± 1     |
| Samsung Internet | 15+            |
| Desktop Chrome   | 最新版 ± 2     |

## 9. テスト実施手順

### 9.1 テスト準備

```bash
# 1. テスト環境セットアップ
npm run test:setup

# 2. テストDB初期化
npm run test:db:reset

# 3. テストサーバー起動
npm run test:server
```

### 9.2 テスト実行

```bash
# P0ブロッカーのみ
npm run test:p0

# 全テスト
npm run test:all

# E2Eテスト
npm run test:e2e

# パフォーマンステスト
npm run test:perf
```

## 10. 合否判定フロー

```mermaid
flowchart TD
    A[テスト実施] --> B{P0全て合格?}
    B -->|No| C[❌ リリース不可<br/>ブロッカー修正]
    B -->|Yes| D{P1合格率 >= 80%?}
    D -->|No| E[⚠️ 条件付きリリース<br/>P1項目をバックログへ}
    D -->|Yes| F{P2合格率 >= 60%?}
    F -->|No| G[✅ リリース可能<br/>P2改善は次回へ}
    F -->|Yes| H[✅ 完全リリース可能]
    C --> I[再テスト]
    I --> A
```
