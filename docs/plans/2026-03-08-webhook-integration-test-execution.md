## 1. テスト概要

外部システムからClaw-Empireへのwebhook接続に関する統合テストを実施する。

**対象エンドポイント**: `POST /api/inbox`

---

## 2. 技術要件統合

### 2.1 開発チーム要件（Aria）

| 項目 | 仕様 |
|:-----|:-----|
| **HTTPメソッド** | POST |
| **コンテンツタイプ** | application/json |
| **認証** | `x-inbox-secret: <INBOX_WEBHOOK_SECRET>` ヘッダー |
| **ペイロード** | JSON形式のwebhookデータ |
| **タイムアウト** | 5秒 |
| **レスポンス** | 200（成功）, 400（不正リクエスト）, 500（サーバーエラー） |
| **署名検証** | HMAC-SHA256 |
| **リトライ処理** | 指数バックオフ |

### 2.2 インフラセキュリティチーム要件（Vault）

| 項目 | 仕様 |
|:-----|:-----|
| **TLSバージョン** | TLS 1.3強制 |
| **送信元IP制限** | ホワイトリスト方式 |
| **レートリミット** | 10 req/min（超過時429） |
| **ポート** | 443のみ |
| **ログ連携** | SIEMへ転送 |

### 2.3 運営チーム要件（Atlas）

| 項目 | 仕様 |
|:-----|:-----|
| **テスト環境** | `/api/webhook/test/*` で分離 |
| **ログ監視** | Winston/Bunyanベース |
| **アラート通知** | Slack/Teams webhook連携 |
| **結果集計** | 自動レポート機能 |

## 3. 受入基準（Acceptance Criteria）

### 3.1 P0 (Critical) - ブロッカー

| ID | テスト項目 | 合格条件 | 失敗条件 |
|:---|:-----------|:---------|:---------|
| **WH-P0-001** | HTTP POST受信 | `POST /api/inbox` が200を返す | 4xx/5xxエラー |
| **WH-P0-002** | 認証検証 | 不正なsecretで401/403が返る | 認証なしで成功 |
| **WH-P0-003** | 署名検証 | HMAC-SHA256で署名検証が動作 | 署名不正でも成功 |
| **WH-P0-004** | タイムアウト処理 | 5秒でタイムアウトし適切なレスポンス | 無限ハング |
| **WH-P0-005** | TLS 1.3強制 | TLS 1.2以下が拒否される | 古いTLSで接続成功 |

### 3.2 P1 (High) - 機能要件

| ID | テスト項目 | 合格条件 | 失敗条件 |
|:---|:-----------|:---------|:---------|
| **WH-P1-001** | リトライ処理 | 指数バックオフで再試行 | 即時再試行または未実装 |
| **WH-P1-002** | レートリミット | 11 req/minで429が返る | 無制限にリクエスト成功 |
| **WH-P1-003** | IPホワイトリスト | 許可IPのみ接続可能 | 不明IPから接続成功 |
| **WH-P1-004** | テスト環境分離 | `/api/webhook/test/*` が本番と分離 | 本番環境に影響 |
| **WH-P1-005** | ログ出力 | 全リクエストがWinstonでログ記録 | ログ欠落 |

### 3.3 P2 (Medium) - 運用改善

| ID | テスト項目 | 合格条件 | 失敗条件 |
|:---|:-----------|:---------|:---------|
| **WH-P2-001** | アラート通知 | 失敗時Slack/Teamsに通知 | 通知未送信 |
| **WH-P2-002** | 自動レポート | テスト結果が自動集計 | 手動集計のみ |
| **WH-P2-003** | SIEM連携 | ログがSIEMへ転送 | 連携未実装 |

## 4. 検証チェックリスト

### 4.1 正常系テスト

- [ ] 有効なsecretでwebhook送信 → 200 OK
- [ ] 正しいJSONペイロードで処理成功
- [ ] TLS 1.3で接続確立
- [ ] 許可IPからのリクエスト成功
- [ ] レート制限内の連続リクエスト成功

### 4.2 異常系テスト

- [ ] 不正なsecret → 401/403
- [ ] 不正な署名 → 401/403
- [ ] 不正なJSON形式 → 400
- [ ] TLS 1.2以下 → 接続拒否
- [ ] 不明な送信元IP → 403
- [ ] レート制限超過 → 429
- [ ] タイムアウト発生 → 適切なエラー処理

### 4.3 リトライ・復旧テスト

- [ ] 一時的エラー時の指数バックオフ再試行
- [ ] 復旧後の正常リクエスト処理
- [ ] リトライ回数上限到達時の処理

### 4.4 負荷テスト

- [ ] 10 req/min 連続送信で正常処理
- [ ] レート制限超過時の429返却
- [ ] 並行リクエスト処理

## 9. リスクと対応策

| リスク | 影響 | 対応策 |
|:-------|:-----|:-------|
| 本番環境への影響 | 高 | テスト環境分離、dry-runモード |
| レート制限の誤検知 | 中 | テスト用IP除外設定 |
| 証明書有効期限 | 中 | 事前確認と自動更新設定 |
| SIEM連携遅延 | 低 | フォールバックとしてファイルログ |
